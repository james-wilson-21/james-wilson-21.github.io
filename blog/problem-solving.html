Problem Solving

You got a problem?

An example from the last sprint we just did was problem solving some of the Kata. In particular the Kata on in-built JavaScript Methods. I wouldn't say I solved them in very elegant ways as there's nothing very elegant about the way I code. I mostly bludgeon away with quick fire solutions until something sticks. In a lot of these cases the tried and true methods weren't cutting the mustard. So I took my time with a lot of them and trialled and errored as well as employed the use of Google and stack overflow to figure out a lot of the solutions. My biggest issues often is going too fast and not checking in with things like variables and scope.

Psuedocode

A programmer looking to deal with a particular problem may run the steps in prose before translating those steps into the targetted programming language.

The rubber ducky method

Explaining the problem to someone, anyone, or anything may and sometimes does help the person resolve the problem once they have explained the problem in it's entirety, out loud. It helps the explainer teach the problem and themselves, more fully understand it..

Reading error messages

Interesting to read messages from the bottom up in Javascript to see where it starts, where our code ends -- where the problem is where the the third party (JS Libraries etc) code is throwing the error message And the error message itself at the top

HackerNoon

Console.logging:

I fully endorse this. It's become my go to for why things aren't working. Maybe there's a better way. If so let me know what it

Googling:

Google here is my exact problem, give me answers from Stack Overflow and all that good stuff. Good way to see who else has encountered this problem in the past and if someone can shine some light on it. Be careful that your problem is actually this, or you could spend hours searching for something that is not your actual problem.
